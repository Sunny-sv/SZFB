# PC 端报表 PRD

## 何时使用

用户要写 PC 端报表/统计/核算类页面 PRD，或已有原型/讨论结论需落文档时启用。
配合 pm-core 工作流：讨论、原型确认后再写 PRD；不擅自改已确认交互。

## 写法风格

必须同时遵守 [pm-prd-writing-style](../pm-prd-writing-style/SKILL.md)：只写有用内容，改动按查询条件、按钮操作、数据列表等模块归类；表格长内容拆有序列表，展示规则等非字段内容放表格最后说明行；语言用日常产品经理口吻。

控件/组件命名与说明列写法遵守 [pm-component-catalog](../pm-component-catalog/SKILL.md)：点名系统已有组件，只写本页业务差异，不重述大家熟悉的交互。

## 与 pc-maintain-prd 的关系

章节 **1～4** 直接遵循 [pc-maintain-prd](../pc-maintain-prd/SKILL.md)：

- 一、概述（版本修订表规则）
- 二、要解决的问题
- 三、产品分析（3.1 产品方案）
- 四、需求分析（需求价值、用户群体、优先级及功能点）

章节 **5 详细需求描述** 起按本 skill 的报表模块模板写。
报表默认**无**新增/编辑/删除/变更记录 Tab；若某 Tab 带维护能力，回退引用 pc-maintain-prd 的按钮与变更记录规则。

## 文档总结构

按顺序输出以下章节（无内容则写「无」或「待补充」）：

1. **概述** — 见 pc-maintain-prd「概述 · 版本修订表」
2. **要解决的问题** — 版本 + 问题描述（侧重人工核算、数据追溯、效率痛点）
3. **产品分析** — 3.1 产品方案（方案层：自动核算、规则匹配、数据同步等能力）
4. **需求分析** — 需求价值、用户群体、优先级及功能点
5. **详细需求描述**
   - 5.1 相关链接（ones、原型）
   - 5.2 权限说明（菜单权限、数据权限）
   - 5.3 需求描述（全局：菜单路径、**数据排序**、**操作提示语**；**不写**原型已可见的页面布局/模块顺序）
   - 5.3.x **各 Tab**（见下方模块模板）

## Tab 层级与类型

多 Tab 报表常见结构（按实际裁剪）：

| Tab 类型 | 定位 | 典型能力 |
|---------|------|---------|
| **明细** | 源数据 + 字段计算 | 定时任务、前置说明、字段表（平铺三列/嵌套四列）、复杂计算规则 |
| **人员/对象汇总** | 按人/单分组汇总 | 分组维度、金额加总口径、双击下钻至明细 |
| **组织/分拨汇总** | 按组织分组汇总 | 由上一层汇总再分组、双击下钻至人员汇总 |
| **终版/调整** | 人工导入覆盖层 | 导入覆盖规则、创建人/时间 |

层级关系：明细 → 汇总 → 更高层汇总；终版 Tab 通常独立，不自动从下钻链路生成。

## 单个报表 Tab 模块模板

每个 Tab 包含 **查询条件 → 按钮操作 → 数据列表** 三节。

### 查询条件

| 名称 | 控件 | 说明 |

说明列必须写清（控件标准名见组件目录）：
- 枚举项与**默认值**
- **空值是否参与过滤**（惯例：空 = 不生效）
- **查询逻辑**（包含 / 交集 / 精准匹配 / 时间区间）
- **查询规则/视角切换**（如有）：枚举、默认值、**影响哪些实体字段参与过滤**
- 时间类：月份区间默认「当月～当月」
- 员工/岗位组件：可选范围（是否限转运人员/岗位）
- 数据权限组件：多选及权限范围

不要复述组件通用交互。片段可复制 [../pm-component-catalog/reference.md](../pm-component-catalog/reference.md) 与本 skill [reference.md](reference.md)。

### 按钮操作

报表默认**两列表** `名称 | 说明`（无变更记录第三列）：

| 按钮 | 必写内容 |
|------|---------|
| **导出** | 导出范围（按筛选条件全量字段）、**文件名格式** `{报表名}_{Tab名}导出_{年月日时分秒}` |
| **下载模版** | 模版文件名（终版/导入 Tab） |
| **导入** | 文件选择方式、字段必填、唯一性校验、覆盖/失败规则、失败文件下载、成功/失败提示语 |

**默认不写**：新增、编辑、删除、终止。

导入校验按三层写：
1. **数据规则校验** — 字段必填、格式、枚举合法性
2. **唯一性校验** — 业务主键组合（如补贴月份 + 分拨 + 工号 + 岗位）
3. **覆盖规则** — 与已有数据的冲突处理（整月覆盖 / 整份失败等）

### 数据列表

按顺序写以下块（无则省略对应块）：

#### A. 定时任务执行规则（明细 Tab 常见）

- **日常频次**：如每天固定 15:00 执行当月数据
- **月度频次**：如每月 1～5 号 15:00 执行上月整月数据

#### B. 前置说明（纳入统计口径）

写在字段表首行或独立编号列表：
- 人员/单据**数据来源**（取自哪个维护模块或主数据）
- **纳入条件**（在职校验、时限、方案匹配前提等；不满足则不出现在报表）
- 与配置/方案表的关联前提

#### C. 字段表

**列数按是否有嵌套字段决定：**

| 场景 | 列 |
|------|-----|
| **无二级字段**（字段均平铺） | `字段 \| 说明 \| 补充说明`（**不要**空着二级字段列） |
| **有嵌套实体**（如新员工/推荐人分组） | `一级字段 \| 二级字段 \| 说明 \| 补充说明` |

| 写法 | 规则 |
|------|------|
| 平铺字段 | 用三列表；字段名写在「字段」列 |
| 嵌套实体 | 用四列表；一级为分组名，二级为子字段 |
| 说明列 | 取值时点（「定时执行时」）、空值展示（`-`）、格式、**双击下钻**（跳转 Tab + 自动带入查询条件） |
| 补充说明列 | 复杂匹配优先级、金额计算公式、为零条件等长逻辑 |

**汇总 Tab** 字段表首行写分组说明：
> 由「{上游 Tab 名}」根据「{分组字段列表}」分组汇总。

汇总金额列须写清：
- 加总字段来源（上游 Tab 的哪个字段）
- 过滤条件（如仅统计某补贴类型、某角色归属的金额）

### 终版/调整 Tab（可选）

与计算层 Tab 的差异：
- 查询条件、列表字段精简（通常无复杂计算列）
- 列表增加**创建人**、**创建时间**（取导入成功的登录人与时间）
- 导入**覆盖规则**须明确（如按补贴月份整月覆盖已有数据）

## 模块继承 / 差异写法

多 Tab 逻辑高度相同时，**不要全文复制**。写法：

> 功能整体与「{基准 Tab 名}」保持一致，仅存在以下差异：
> 1. …
> 2. …

差异点限定在：查询项、分组维度、汇总口径、导入模版名、覆盖规则、下钻目标等。

## 写作原则（报表专属）

- 原型已确认时，**不写**页面结构复述（有无 Tab、查询/按钮/列表顺序、通用三区布局等）；5.1 挂原型链接，只写原型看不出来或需约定的规则
- 字段/状态名与原型、现有系统保持一致；不确定标「待确认」
- 计算/匹配规则用**优先级逐级匹配，命中即终止**的写法
- 金额为零条件写「满足以下**任一**条件则金额为 0」
- 分拨维度 / 全国维度 / 人员维度校验**分开写**，不要混写
- 各种操作的**提示语**原样写出
- 注意描述**数据排序规则**
- 取值时点写清「定时执行时」vs「查询时」，避免口径歧义

## 产出前自检

- [ ] 章节 1～4 遵循 pc-maintain-prd 规则（版本表、修订人胡绵飞、当天日期）
- [ ] 5.3 已写菜单路径、数据排序、操作提示语规范
- [ ] 每个查询条件有默认值 + 空值是否生效 + 查询逻辑
- [ ] 导出文件名格式已写明（每个 Tab）
- [ ] 明细 Tab：定时任务（若有）、前置说明、字段表完整（平铺用三列，有嵌套用四列）
- [ ] 计算/匹配规则：优先级清晰，未命中时的兜底结果已说明
- [ ] 汇总 Tab：分组维度、加总口径、双击下钻联动已写
- [ ] 终版 Tab：导入三层校验 + 覆盖规则 + 创建人/时间（若有）
- [ ] 按钮表为两列；无编造的新增/编辑/删除（除非用户明确要求）
- [ ] 多 Tab 差异用继承写法，无大段重复
- [ ] 未复述原型已可见的页面布局（模块顺序、有无 Tab 等）
- [ ] 业务计算细节来自讨论/原型；未确认处标「待确认」

## 交付前必审

写完本报表 PRD 后，**先启用** [pm-review](../pm-review/SKILL.md) 做逻辑/结构/写法审查，输出「待你确认」清单后再结束本轮（用户明确说「先别审」除外）。默认只提问题，不擅自改文档。
