# PRD 写法规范

## 何时使用

- 写维护页、配置页、报表页 PRD 时
- 写已有功能新增、修改、版本增量说明时
- 用户说「只写改动」「不要废话」「按模块写」「AI 味不要太重」「原来那种写法」时
- 与 `pc-maintain-prd`、`pc-report-prd`、`pm-incremental-prd` 同时使用，本 skill 管写法风格
- 控件叫什么、说明列写差异不写常识：见 [pm-component-catalog](../pm-component-catalog/SKILL.md)

## 核心原则

|原则|要求|
|---|---|
|只写有用内容|只写本次新增、删除、调整的内容；不展开没改的内容|
|按模块归类|查询条件的改动放查询条件；按钮的改动放按钮操作；列表的改动放数据列表|
|不跨模块重复|维护页已说明、已禁止或大家评审时已确认的内容，计算/报表/其他文档不再重复写|
|删除就写删除|不要写「替代为」「由 A 替代」「同步更新配置项」「配置内容升级」|
|表格内分点|同一单元格里有多类信息时，用 `1.` `2.` `<br>` 拆开|
|非字段不装字段|展示规则、合并规则、排序规则等放表格最后说明行，不放字段列|
|不重复原型已见内容|原型已确认时，不写页面骨架、模块顺序、有无 Tab、查询/按钮/列表三区布局等界面复述；5.1 已挂原型链接即可|

## 禁止写法

- 不写「本版没改什么」清单。
- 不写「其余同 V1.0.0」这类兜底废话。
- 不单独写「删除项」总表；删除内容放回对应模块。
- 不把基线版本全文复制一遍。
- 不重复别的文档已经讲清楚的配置项、字段枚举、页面限制。
- 不写“因为维护页已经禁止，所以计算侧也不匹配”这类自然成立的推导。
- 不写「查询条件 → 按钮 → 列表」等页面模块顺序（原型已确认时）。

## 推荐结构

增量或修改类 PRD 优先按模块写：

1. **本版改动概述**：3～8 条，短句说明范围。
2. **查询条件**：只写新增、删除、调整的查询项。
3. **按钮操作**：只写新增、编辑、导入、导出等本次变化。
4. **数据列表**：只写新增、删除、调整的字段。
5. **变更记录 / 验收要点**：只写本次涉及的内容。

## 跨文档取舍

- 维护页负责写：字段、枚举、控件、导入模板、页面置灰/禁用、列表展示规则。
- 计算逻辑负责写：运行时真正会用到的匹配、取值、命中、兜底、落库。
- 报表负责写：查询、统计口径、字段取值、汇总/下钻、导出。
- 如果某个限制已经在维护页禁止维护，计算逻辑里不要再写“禁止后自然匹配不到”。
- 如果前文已经列过字段取值表，后文不要重复表格，也不要写「见上文」；直接写动作或删掉该步。

## 表格写法

### 查询条件

长内容要拆开：

```markdown
|名称|说明|
|---|---|
|组合项|1. **删除**：「操作结算重量段」「体积段」。<br>2. **新增**：票-操作结算重量段、票-体积段、票-件数段。<br>3. **查询逻辑**：不变。|
```

### 按钮操作

按钮说明按动作分层，不把多件事塞一句：

```markdown
|名称|说明|针对变更记录|
|---|---|---|
|导入|1. **不支持导入**：按区间配置方案。<br>2. **支持导入**：不区分方案。<br>3. **拒收列**：区间维度、匹配维度、区间分列。|—|
```

### 数据列表

字段表只放字段；展示规则放最后一行。

**列数：**无二级字段时用 `字段 | 说明 | 补充说明` 三列；有嵌套分组时才用 `一级字段 | 二级字段 | 说明 | 补充说明` 四列，**不要**平铺字段也留空的二级字段列。

```markdown
|字段|说明|补充说明|
|---|---|---|
|分拨中心|算薪分拨|定时执行时写入|
|展示规则||1. 金额右对齐。<br>2. 无值展示 `-`。|
```

有嵌套时见 `pc-report-prd` 四列表写法。

## 语言风格

- 像产品经理写给研发/测试看的说明，不像宣传稿。
- 版本说明用日常说法：`这次主要是把配置方式改了` 可以，`配置内容升级` 少用。
- 复杂规则可以写清楚，但不要故意写长。
- 不确定的业务口径标「待确认」，不要替业务拍板。

### 通俗易懂（与 `pm-core` 一致）

- 不用研发/数据侧黑话：存量、刷数、映射、兜底、运行时、递推等 — 换成业务能看懂的说法。
- 比较关系用中文：**不低于、低于、达到、不超过**；不用 ≥、<、OR、AND。
- 组合条件写清楚：**满足其中一个即可** / **两个都要满足**，不要写「OR 组合」「AND 组合」。
- 取数范围用例子：**算 7 月工资，连续看 2 个月，就看 5 月、6 月**，少写 M-N…M-1 这类公式。
- 功能未上线、没有历史数据时，直接写「本功能未上线」，不写「存量方案兜底」「待研发确认映射」。

**示例：**

| 不推荐 | 推荐 |
|---|---|
| 升档：绩效分均 ≥ 门槛 | 升档：每个月绩效分都不低于门槛 |
| 满足其一 OR 组合 | 满足其一：无破损或绩效分达标，有一个就行 |
| 无存量落库需刷数 | 本功能未上线，按新字段维护即可 |

### 跨表 / 跨报表取数

从另一张表、报表取标准或配置时，**写成一句业务话**：用什么条件匹配哪张表 → 本字段取对方哪个字段 → 未匹配到怎么办。

- 不要拆成「关联键」小标题 + 字段对照表 +「未关联到时」三截，除非字段很多、一句写不下。
- 匹配条件只列实际用到的维度即可；不必再写「不含某某字段」——没写进匹配条件的，本来就不参与匹配。
- 用「匹配」「取」「未匹配到」；少用「关联键」「字段映射」「关联失败兜底」。

**示例：**

| 不推荐 | 推荐 |
|---|---|
| 关联键：分拨 + 工号 + 岗位 + 月份（不含是否支援）。绩效工资对应报表「保底绩效」；未关联到时标准按 0。 | 根据分拨中心 + 人事工号 + 岗位 + 结算月份匹配【机叉叉车组长绩效标准报表】，取「保底绩效」；未匹配到时标准为 0。 |

### 字段计算写进说明列

算薪、提成等**字段取值逻辑**，写进该字段「说明」单元格（有序列表：匹配 → 取值 → 门槛 → 公式），不要在字段表下另起「从报表取标准」「实发计算」等小节。

- 每个改动字段的说明**当场写全**，不写「见下文」「按下方规则」「门槛见上文」。
- 两个字段规则相同，也各自写全；不要用「同某某字段」互相指。
- 表下只留展示、汇总、导出等**非单字段计算**的内容。

**示例结构（说明列内）：**

```markdown
1. **标准**：根据 … 匹配【××报表】，取「××」；未匹配到时标准为 0。
2. **绩效分**：根据 … 取【绩效分导入】对应岗位 tab；未取到则本字段为 0。
3. **发放门槛**：… → 本字段为 0。
4. 门槛满足：本字段 = 标准 × 绩效分 / 100 × 出勤系数。
```


## 输出前自检

- [ ] 是否按模块归类，而不是单独堆「删除项」？
- [ ] 没改的内容是否没有展开？
- [ ] 是否没有重复其他模块已说明、已禁止或已评审的内容？
- [ ] 表格长句是否拆成有序列表？
- [ ] 展示规则、排序规则是否放在表格最后说明行？
- [ ] 原型已确认时，是否未复述页面布局、模块顺序等界面骨架？
- [ ] 是否避免「替代为」「同步更新配置项」等 AI 腔？
- [ ] 是否无 ≥/<、OR/AND、存量、刷数、映射等研发向用语（见 `pm-core` 写法约定）？
- [ ] 跨表取数是否写成「匹配条件 + 取哪个字段 + 未匹配怎么办」一句，而不是「关联键」拆段？
- [ ] 字段计算逻辑是否写进该字段说明列，表下没有另起「取标准 / 实发计算」小节？
- [ ] 字段说明是否当场写全，没有「见下文」「同某某字段」？

完整逻辑/结构/原型对照审查用 [pm-review](../pm-review/SKILL.md)，不单靠本清单。
