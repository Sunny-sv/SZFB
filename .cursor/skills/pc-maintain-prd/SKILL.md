# PC 端维护页 PRD

## 何时使用

用户要写 PC 端维护/配置类页面 PRD，或已有原型/讨论结论需落文档时启用。
配合 pm-core 工作流：讨论、原型确认后再写 PRD；不擅自改已确认交互。

## 写法风格

必须同时遵守 [pm-prd-writing-style](../pm-prd-writing-style/SKILL.md)：只写有用内容，改动按查询条件、按钮操作、数据列表等模块归类；表格长内容拆有序列表，展示规则等非字段内容放表格最后说明行；语言用日常产品经理口吻。

控件/组件命名与说明列写法遵守 [pm-component-catalog](../pm-component-catalog/SKILL.md)：点名系统已有组件，只写本页业务差异，不重述大家熟悉的交互。

## 文档总结构

按顺序输出以下章节（无内容则写「无」或「待补充」）：

1. **概述** — 版本修订表，填写规则见下方「概述 · 版本修订表」
2. **要解决的问题** — 版本 + 问题描述
3. **产品分析**
   - 3.1 **产品方案** — 按版本描述方案，表格列：**版本 | 产品方案**
4. **需求分析** — 需求价值、用户群体（岗位/部门/频次）、优先级及功能点
5. **详细需求描述**
   - 5.1 相关链接（ones、原型）
   - 5.2 权限说明（菜单权限、数据权限）
   - 5.3 需求描述（全局注意事项，如排序规则、提示语规范、菜单路径）
   - 5.3.x **各 Tab/模块**（见下方模块模板）
   - 若有变更记录 Tab，单独一节

## 概述 · 版本修订表

表格列：版本 / 修订人 / 修订时间 / 修订说明 / 开发负责人 / 测试负责人

| 字段 | 规则 |
|------|------|
| **版本** | 新功能默认 **V1.0.0**；后续改动按幅度递增：**V1.0.1**（小修小补、文案/字段微调）、**V1.1.0**（功能增强、新增模块/按钮/查询项）、**V2.0.0**（架构或业务逻辑大改、不兼容变更） |
| **修订人** | 固定 **胡绵飞** |
| **修订时间** | 取**当前日期**，格式 **YYYY.M.D**（到年月日即可，如 2026.6.29） |
| **修订说明** | **总结本版功能点**写入，一句话或分点概括本次交付范围 |
| 开发负责人 / 测试负责人 | 无信息则留空 |

后续版本在表中**追加新行**，保留历史版本记录，不覆盖旧行。

## 产品分析 · 产品方案

小节标题固定为 **3.1 产品方案**，用表格按版本写方案摘要：

| 版本 | 产品方案 |
|------|---------|
| V1.0.0 | {方案描述：基于什么场景、通过什么能力、达成什么效果} |

- 与「要解决的问题」呼应，写**方案层**而非细节需求
- 版本号与概述表保持一致
- 后续改动追加新行

## 单个维护模块模板

每个 Tab 或功能块包含 **查询条件 → 按钮操作 → 数据列表** 三节，均用 Markdown 表格。

### 查询条件

| 名称 | 控件 | 说明 |

控件列用组件目录标准名（如数据权限组件、人员组件、多选下拉框）。说明列必须写清：
- 枚举项与**默认值**
- **空值是否参与过滤**（惯例：空 = 不生效）
- **查询逻辑**（包含 / 交集 / 精准匹配 / 时间区间交集）
- 联动与置灰规则（如有）

不要复述组件通用交互（可搜索、弹层多选等）。查询区布局约定：默认展示行数、展开规则（若适用）单独一句说明。

### 按钮操作

**无变更记录 Tab 时**（默认，如业务参数配置等标准维护页）：

| 名称 | 说明 |

**有变更记录 Tab 时**（如货量重量配置）：

| 名称 | 说明 | 针对变更记录 |

「说明」列按层次写：
1. **操作限制**（选中条数、状态、权限）
2. **弹框/页面** — 分区写：基础设定、匹配条件配置、业务参数等
3. 每个字段：**控件类型、枚举、默认值、必填、联动/置灰、输入限制**
4. **确定/提交校验** — 唯一性/重复校验、提示弹框
5. **提示文案** — 原样写出

「针对变更记录」列：**仅在有变更记录 Tab 时写出**第三列；写保存成功后变更 Tab 的联动（修改类型、有效周期、历史记录截断规则）。**无变更记录 Tab 时不写该列、不单独说明。**

常见按钮按需选用：新增、编辑、批量编辑结束时间、**删除**、导出、下载模版、导入。

**删除 vs 终止**：
- **默认写「删除」**：操作限制、确认提示（如「删除后不可恢复，请确认是否删除！（共 X 条）」）、删除后列表行为
- **特殊页面写「终止」**：仅用于重要规则/配置类、**不允许物理删除**的数据；需写清结束时间、状态、优先级重排等
- 用户未说明时，**优先用删除**；提到「不可删」「只能终止」「规则留痕」时再改用终止

**简单维护页**（无变更记录、无时效规则）：参照业务参数配置类 — 编码唯一性校验、编辑时主键置灰、导入整份成功/失败即可，不必写分拨纬度/有效期限等复杂逻辑。

### 数据列表

| 字段 | 说明 | 双击事件 |

说明列写：取值来源、空值展示（如「-」）、格式、状态判定、展示样式。
双击事件：跳转、弹窗、切换 Tab 等；无则留空。

## 变更记录 Tab（可选）

**仅当页面有审计/追溯需求时**才单独写一节，此时按钮操作表使用三列（含「针对变更记录」）。

结构：
- 查询条件（变更类型、方案编号、有效周期、变更人/时间等）
- 按钮（通常仅导出；按钮表为两列 `名称 | 说明`）
- 数据列表 + **变更对比弹窗**（变更前/后列，变更字段标色）

主列表双击勾选行 → 跳转变更记录 Tab 并自动查询，为常见联动，按需描述。

## 模块继承 / 差异写法

多 Tab 逻辑高度相同时，**不要全文复制**。写法：

> 功能整体与「{基准 Tab 名}」保持一致，仅存在以下差异：
> 1. …
> 2. …

差异点限定在：枚举值、字段名联动、导入模版名、校验范围等。

## 写作原则

- 写**用户可见行为与系统表现**，不写表结构、接口协议、框架选型
- 字段/状态名与原型、现有系统保持一致；不确定标「待确认」
- 每个操作写清：**正常路径 + 校验失败 + 边界**（如整份导入失败）
- 各种操作的**提示语**单独写出，不省略
- 注意描述**数据排序规则**
- 分拨纬度 / 全国纬度 校验逻辑分开写，不要混写

## 产出前自检

- [ ] 概述版本表：版本号规则正确、修订人为胡绵飞、修订时间为当天日期、修订说明已概括功能点
- [ ] 产品分析已含 3.1 产品方案表（版本 | 产品方案），与「要解决的问题」呼应
- [ ] 每个查询条件都有默认值 + 空值是否生效 + 查询逻辑
- [ ] 每个按钮都有操作限制 + 弹框字段 + 提交校验 + 提示语
- [ ] 列表字段有取值来源与展示规则
- [ ] 导入/导出/模版文件名已写明（若有）
- [ ] 按钮操作表：无变更记录 Tab 用两列；有变更记录 Tab 用三列
- [ ] 变更记录：有 Tab 则单独一节 + 第三列联动；**无 Tab 则不写、不单独说明**
- [ ] 多 Tab 差异用继承写法，无大段重复

## 交付前必审

写完本页 PRD 后，**先启用** [pm-review](../pm-review/SKILL.md) 做逻辑/结构/写法审查，输出「待你确认」清单后再结束本轮（用户明确说「先别审」除外）。默认只提问题，不擅自改文档。
