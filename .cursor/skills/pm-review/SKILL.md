# PRD / 原型审查

## 何时使用

- **交付前必审**：刚用 `pc-maintain-prd` / `pc-report-prd` / `pm-incremental-prd` / `pc-prototype` 写完时，**先跑本审查、输出「待你确认」清单，再结束本轮**（除非用户明确说「先别审」）
- 用户说「审查 PRD / 检查原型 / 挑挑刺 / 写完帮我过一遍 / PRD 自检 / 原型评审」
- 评审前想确认：逻辑有没有打架、结构合不合标准、原型展示/交互有没有问题

**不替代：** 写作用对应 PRD/原型 skill；讨论补方案用 `pm-discuss-challenge`。本 skill 只管**审已有产出物**。

## 红线

| 禁止 | 要求 |
|------|------|
| 静默改文档 | 默认**只输出问题清单**；改文件须用户明确说「按建议改 / 改掉第 N 项」 |
| 建议写成已定 | 不写「已修复」「已按标准改好」除非本轮真的改过且用户同意 |
| 编造冲突 | 不拿未确认的公司内部规则当「冲突」依据 |
| 装成事实 | 拿不准标「可能冲突 / 待确认」，严重度用「提醒」 |

## 工作流（固定顺序）

1. **确定审查对象**
   - 用户指定了路径 → 用该文件
   - 本轮刚写出的 PRD/原型 → 用刚产出的文件
   - 未指定且上下文没有 → 先问路径（PRD / 原型 / 两者对照）
2. **读目标文件**（PRD Markdown 和/或 `docs/**/*.html` 业务原型）
3. **按类型读标准**（只读需要的，见下表）；详细检查项读 [checklist.md](checklist.md)
4. **过一遍 checklist**，只记真实问题，不凑条数
5. **按下方模板输出**；无问题也写「未发现冲突 / 结构与标准一致」

### 按类型对照哪些标准

| 审什么 | 必须对照 |
|--------|----------|
| 维护页 PRD | [pc-maintain-prd](../pc-maintain-prd/SKILL.md) + [pm-prd-writing-style](../pm-prd-writing-style/SKILL.md) + [pm-component-catalog](../pm-component-catalog/SKILL.md) |
| 报表 PRD | [pc-report-prd](../pc-report-prd/SKILL.md) + writing-style + component-catalog |
| 增量/规则 PRD | [pm-incremental-prd](../pm-incremental-prd/SKILL.md) + writing-style |
| HTML 原型 | [pc-prototype](../pc-prototype/SKILL.md) + 母版 `docs/_共用/PC端通用框架.html` |
| 两边都有 | 上表对应项 + checklist「原型 ↔ PRD」对齐 |

分不清文档类型时：看章节（有「查询条件/按钮操作/数据列表」多为维护或报表；只写本版改动多为增量；`.html` 在 `docs/{项目名}/` 为原型）。

## 输出模板

```markdown
## 审查结论
一句话：通过 / 有 N 项待确认（含 X 阻断）

## 问题清单（待你确认）
| 编号 | 严重度 | 类型 | 位置 | 问题 | 建议 |
|------|--------|------|------|------|------|
| 1 | 阻断 / 建议 / 提醒 | 逻辑冲突 / 结构 / 写法 / 原型布局 / 交互 / 原型-PRD不一致 | 章节或文件位置 | … | … |

## 已通过项（简略）
- [ ] 仅列关键通过项，不写长篇表扬
```

无问题时，「问题清单」写「无」；「已通过项」可列 3～6 条关键打勾项。

### 严重度

| 级别 | 何时用 | 对用户的含义 |
|------|--------|--------------|
| **阻断** | 逻辑互相打架、缺必填结构模块、原型三区/四模块骨架被改坏 | 建议先改再拿去评审 |
| **建议** | 写法不合规范、交互演示缺失、易误解 | 你决定改不改 |
| **提醒** | 行业常见坑、口径需业务拍板、可能冲突但证据不足 | 标待确认，不替你定 |

### 类型取值（表内只用下列之一）

`逻辑冲突` · `结构` · `写法` · `原型布局` · `交互` · `原型-PRD不一致`

## 写法约定

- 结论先行；问题可直接回答「改 / 不改 / 改成…」
- 正文中文；位置写清章节名或原型模块（筛选区/按钮区/列表列名等）
- 建议要可执行（改哪一句、补哪张表），不要空泛「建议优化」
- 一次审查优先列真正影响评审的项；同一根因合并为一条
