# PRD / 原型审查检查项

审的时候按审查对象勾选对应节；不必全文复述标准 skill，有疑点再打开对应 SKILL。

---

## 一、PRD 逻辑

- [ ] 同一字段/规则前后是否矛盾（默认值、空值是否过滤、匹配条件、未命中怎么办）
- [ ] 同一枚举/状态在不同章节写法是否一致
- [ ] 维护页已禁止维护的项，计算/报表是否仍假设可以配、可以命中
- [ ] 导入拒收规则与列表可展示/可编辑是否打架
- [ ] 权限说明与按钮/数据可见范围是否矛盾
- [ ] 增量文档是否出现「见上文」「同某某字段」「见 X.X / 第 X 章」等互引（应当场写全）
- [ ] 跨表取数是否写清：匹配条件 → 取哪个字段 → 未匹配怎么办；两边口径是否同一套
- [ ] 生效/失效、跨月切点、覆盖规则等时效逻辑是否自洽（若有）

---

## 二、PRD 结构

### 维护页（对照 pc-maintain-prd）

- [ ] 是否有：概述（版本修订表）→ 要解决的问题 → 业务分析（含 3.1 用户调研、3.2 产品方案，两项都要写）→ 详细需求
- [ ] 详细需求是否含：相关链接、权限、全局注意事项
- [ ] 各 Tab/模块是否按 **查询条件 → 按钮操作 → 数据列表**（有变更记录则单独一节）
- [ ] 维护页是否把新增/编辑弹框写进按钮说明列，而不是单独「新增弹框」「编辑弹框」小节
- [ ] 版本表：修订人、日期格式、修订说明是否按规范；版本号与产品方案表是否一致；用户调研是否有本版内容（无则须写明「无」及原因）

### 报表（对照 pc-report-prd）

- [ ] 章节 1～4 是否与维护页总结构一致（版本表等）
- [ ] 各 Tab：查询、导出、列表、定时任务/汇总下钻/终版导入等应有模块是否齐全（无则应写「无」或说明不适用）
- [ ] 明细/汇总/终版差异是否用继承写法，避免大段复制又改漏

### 增量（对照 pm-incremental-prd）

- [ ] 是否只写本版改动，未整份复制基线
- [ ] **要解决的问题、用户调研、产品方案、需求价值**是否都有本版内容，且问题有具体业务场景（非空话）；用户调研与产品方案是否齐全（无调研须写明「无」及原因）
- [ ] 是否出现「本版改动概述 / 版本概述」总述小节（应删除，规则直接写在模块里）
- [ ] 多版本串联文档：是否**未回改**历史 `### Vx.y.z` 章节原文；本版规则是否只出现在新建版本节 + 概述版本表
- [ ] 改动是否按模块归类（查询/按钮/列表/规则主题），而非单独堆「删除项」总表
- [ ] 单读本文是否能审本版范围（禁止靠「与上一版一致，略」糊弄）

---

## 三、PRD 写法（对照 pm-prd-writing-style）

- [ ] 是否只写有用改动；没改的内容没有展开
- [ ] 表格长内容是否拆成有序列表；展示/排序等非字段规则是否在表格最后说明行
- [ ] 原型已确认时，是否复述了页面骨架、模块顺序、有无 Tab 等（应避免）
- [ ] 是否出现 AI 腔：「替代为」「同步更新配置项」「配置内容升级」等
- [ ] 是否出现研发向用语：存量、刷数、映射、≥/<、OR/AND 等（应换成业务说法）
- [ ] 字段计算是否写进该字段说明列且当场写全；有无「见下文」
- [ ] 控件是否点名组件目录标准名，是否只写本页业务差异（对照 pm-component-catalog）

---

## 四、原型展示（对照 pc-prototype + 母版）

- [ ] 是否从 `docs/_共用/PC端通用框架.html` 复制而来（非从零发明布局）
- [ ] **全局结构**完整：顶栏（品牌区 + 蓝底标签栏）+ 侧边栏 + 主内容区
- [ ] **四模块**顺序：筛选条件 → 内容 Tab → 操作按钮 → 数据列表（含分页）
- [ ] 分页是否贴在内容卡片底部（数据少时也不紧贴表体悬在半屏）
- [ ] 筛选：名称左 + 控件右；默认一行 3 组件；重置/查询在最后一行右侧且底对齐
- [ ] 弹框/表单：字段名与控件左右排列（禁止名称在上、控件在下）
- [ ] 表格：首列复选框；表头与单元格默认居中
- [ ] 单文件 HTML，无 CDN、无后端、无真实 API
- [ ] 页面无「原型说明」黄条、演示备注、临时演示按钮；看起来就是实际业务页
- [ ] 未确认字段是否用通用占位，而未乱编业务名/规则
- [ ] 色值与母版大体一致（顶栏蓝 `#0659d9`、主色、侧栏、内容底、卡片、边框）

---

## 五、原型交互

- [ ] 母版演示交互是否还在：顶栏汉堡折叠侧栏（`#layout.sidebar-collapsed`）、顶栏/内容 Tab 切换 active、表头全选联动行 checkbox
- [ ] 是否误做了查询、重置结果、分页跳转、菜单路由、表单校验等（线框阶段应克制）
- [ ] 若有多 Tab：切换后列表/按钮差异是否看得见、说得清
- [ ] 若有弹层/抽屉：打开与关闭路径是否完整（演示级即可）
- [ ] 关键操作路径是否断（按钮在但无对应演示态，且 PRD 已依赖该路径时标提醒）

---

## 六、原型 ↔ PRD 对齐（两边都有时必做）

- [ ] 筛选条件名称、控件类型是否一致
- [ ] 操作按钮名称、是否存在是否一致
- [ ] 列表列名、Tab 名是否一致
- [ ] PRD 有、原型无 → 列出；原型有、PRD 无 → 列出
- [ ] 权限/菜单路径与原型侧栏菜单文案是否明显冲突（有则标）

---

## 七、产出提醒

- 只把**未通过或存疑**的项写入审查表；通过项可在「已通过项」简略打勾
- 一条问题写清：位置 + 现象 + 建议改法
- 逻辑拿不准时用严重度「提醒」，不要升成「阻断」吓唬人
