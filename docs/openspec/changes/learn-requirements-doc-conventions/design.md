## Context

仓库路径 `docs/` 存放数智分拨产品线需求文档，主题集中在：转运人员薪资、提成方案维护、排班维护、货量报表、机叉/叉车组长绩效等。文档为 Markdown，面向运营、研发、测试、实施，非代码仓库。

当前状态：
- 约 10 份业务 PRD / 摘要 / 会议记录，按业务域分文件夹（如 `机叉:叉车组长绩效需求/`）或放根目录
- 存在**两套章节骨架**（见 spec），内容质量一致、编号习惯不同
- `openspec/config.yaml` 仅有 schema 声明，无项目上下文
- 产品经理工作流（讨论 → 原型 → PRD）在 Cursor rules 中已定义，但未与仓库文档格式绑定

## Goals / Non-Goals

**Goals:**

- 从现有文档**归纳**可复用的写作规范，写入 OpenSpec spec + `config.yaml` context
- 明确文档类型分工：完整 PRD、结论摘要、对接/确认记录
- 为 AI 提供可执行的章节模板、表格范式、版本与交互描述规则
- 保持与现网文档兼容，新文档默认采用**主骨架 A**（见 spec）

**Non-Goals:**

- 不批量重写历史 PRD
- 不定义研发技术实现（表结构、接口、框架）
- 不替业务拍板库存/计费/合规等未决规则
- 不在本 change 内新建对外客户材料

## Decisions

### 1. 规范来源：归纳而非发明

**决策**：以 `转运人员排班维护.md`、`交接单人员提成.md`、`百分比重量涉及流水线货量报表改动.md` 为主样本；`【2508】数智分拨` 系列为**兼容骨架 B**。

**理由**：主骨架 A 版本表字段更全（含开发/测试负责人），且「按版本增量描述问题/价值」的模式在迭代文档中覆盖最广。

**备选**：强制统一为骨架 B 的「文档综述」体例 — 放弃，改动面大且与多数新文档不一致。

### 2. 规范落地位置

**决策**：双写 — (1) `openspec/changes/.../specs/requirements-doc-conventions/spec.md` 作为正式需求；(2) `openspec/config.yaml` 的 `context` 字段放精简版，供 CLI 注入 AI。

**理由**：spec 可归档、可 diff；config context 是生成 artifact 时的即时约束。

### 3. 版本号语义

**决策**：采用 `V{major}.{minor}.{patch}`（如 V8.2.0）；大版本表示新能力块，小版本表示规则调整，补丁表示文案/边界修正。

**理由**：与仓库现网一致。

### 4. 详细需求描述的层级

**决策**：功能页 → Tab（如有）→ 查询条件 / 按钮 / 弹框 / 数据列表；计算类需求可另起「计算规则」小节。

**理由**：与维护页类 PRD 的实际结构一致，研发测试可按表逐项验收。

## Risks / Trade-offs

| 风险 | 缓解 |
|------|------|
| 两套骨架并存导致 AI 混用 | spec 标明默认 A，B 仅用于历史/大模块延续 |
| 规范过细限制表达 | 交互/异常用「编号列表 + 表格」为 MUST，叙述性分析保持 SHOULD |
| config context 过长 | 控制在约 200 行内，细节留在 spec 文件 |
| 飞书/墨刀链接失效 | 规范要求写清链接类型与占位，不强制可访问性 |

## Migration Plan

1. 完成本 change 的 spec、design、tasks
2. `/opsx:apply` 更新 `openspec/config.yaml`（及可选 `openspec/specs/` 归档）
3. 新 PRD 从主骨架 A 起稿；改旧文档时仅追加版本块，不重排历史章节
4. 无需回滚策略（无生产系统变更）
