# 二、要解决的问题

|**版本**|**问题**|
|---|---|
|V1\.3\.0|维护数据作用于考勤核算；员工当月中途离职后，发现配置有误或需新增配置时，因人员维度仅可选在职员工，无法对离职人员维护规则，影响当月考勤核算。|

# 三、业务分析

## 3\.1  用户调研

|**版本**|**调研结果**|
|---|---|
|V1\.3\.0|本维护页配置的数据实际作用于考勤核算。业务方反馈典型场景：某员工当月仅在职半个月后即离职；在其离职后发现为其配置的维护数据有误，或需为其新增配置，但系统人员维度仅允许选择在职员工，导致已离职人员无法查询、新增、编辑及导入，当月考勤核算所需规则无法及时修正。<br>调研结论：人员组件应放开离职限制，使查询与写操作口径一致；导入时以人事工号是否存在为校验边界，不因离职拦截。|

# 四、需求分析

## 4\.1 需求价值

|**版本**|**需求价值**|
|---|---|
|V1\.3\.0|1. **保障考勤核算准确性**：维护数据直接作用于考勤，支持对当月中途离职人员修正或补录规则，避免因选不到离职人员导致核算依据错误。<br>2. **缩短问题处理周期**：离职后发现问题可立即在系统内维护，无需绕行线下或找研发处理。<br>3. **改动范围可控**：仅调整人员可选范围及导入校验，不改变维度、有效月份、重复校验等既有逻辑。|
